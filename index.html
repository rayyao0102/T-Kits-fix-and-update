POWERFUL CAPABILITIES

Core Features

Discover the flexibility and power T-Kits brings to your Minecraft server administration.

Advanced Kit System

Unlimited and Global Kits: Each player can save up to 7 kits (configurable), with support for both private and global kits.
Animated GUI: Modern, themeable GUIs for kit selection, editing, and previewing, with drag-and-drop and live item previews.
Kit Sharing: Share kits with unique codes, import/export, and global browsing.
Ender Chest Integration: Each kit can have linked Ender Chest contents.
Kitroom: Server-wide item repository with categories, cooldowns, and permissions.
Full Validation: Prevent forbidden items, enforce item limits, and require confirmation for destructive actions.

Comprehensive Command Suite

Player Commands: /kit, /k1–/k7, /regear, /arrange, /kit import.
Admin Commands: /tkitsadmin for kitroom management, migration, reload, cooldown reset, and more.
Permissions: Fine-grained permission nodes for every action, kit, and admin operation.
Cooldowns & Combat Tag: All commands respect configurable cooldowns and combat tag restrictions.

Player Data & Kit Management

Async Data Loading: Player data is loaded and saved asynchronously for zero lag.
Safe Caching: In-memory cache with thread-safe access and auto-save on shutdown.
Kit Tracking: Remembers last loaded kit, supports per-player and global kit operations.
Migration Tools: Seamless migration between YAML and MySQL storage backends.

Security & Anti-Exploit

Combat Tag System: Prevents kit loading, regear, and command usage during combat.
Command Blocking: Configurable list of blocked commands during combat.
Permission Checks: All actions are permission-gated, with bypass options for admins.
Validation: Prevents saving forbidden items, enforces slot and item limits, and validates all user input.

Performance & Reliability

Async IO: All storage operations (YAML/MySQL) are fully asynchronous.
Smart Caching: Guava caches for combat tags, cooldowns, and player data.
Resource Management: HikariCP for MySQL pooling, single-threaded YAML IO for safety.
Auto-Recovery: Fallback to YAML if MySQL fails, with error logging and rollback.

Developer API & Integration

Public API: Exposes TKitsAPI for kit management, regear, arrange, and combat tag.
PlaceholderAPI: Built-in expansion for kit and combat placeholders.
Extensibility: Clean interfaces for storage, config, and event hooks.
JavaDocs & Examples: All public methods are documented, with usage examples in the docs section.

Installation Guide

Getting started with T-Kits is simple. Follow these steps.

  1. 1

    Download Plugin

    Get the latest T-Kits version from our website or SpigotMC. Ensure compatibility with your server (1.21+).

    Download Page
  2. 2

    Upload JAR File

    Place the downloaded `T-Kits.jar` file into your server's `/plugins` directory.

    /plugins/T-Kits.jar
  3. 3

    Restart Server

    Restart your Minecraft server to load the plugin and generate configuration files.

    /restart   # or use your panel
  4. 4

    Configure (Optional)

    Adjust settings in `/plugins/T-Kits/config.yml` and other generated files as needed.

    /plugins/T-Kits/config.yml
  5. 5

    Activate License (Premium)

    For Premium, activate your license key to unlock all features.

    /tkits license YOUR-KEY-HERE
  6. 6

    Enjoy T-Kits!

    The plugin is ready! Explore features using the main command.

    /tkits help

Documentation

Commands Reference

Every command, subcommand, and permission in T-Kits is documented below. All commands support tab completion and context-aware suggestions.

/kit
Open the main kit selection GUI.
Permission: tkits.use
/kit import
Import a kit using a share code.
Permission: tkits.kit.share
/k1 – /k7
Load Kit 1–7 directly.
Permissions: tkits.load.1 ... tkits.load.7
/regear or /rg
Gives a special shulker box to regear your inventory.
Permission: tkits.regear
/arrange or /ag
Arranges your inventory according to your last loaded kit.
Permission: tkits.arrange
/tkitsadmin or /tka
Root admin command. Permission: tkits.admin
/tkitsadmin kitroom save
Save Kitroom categories to kitroom.yml.
/tkitsadmin kitroom reload
Reload Kitroom from kitroom.yml.
/tkitsadmin kitroom resetdefaults
Generate default Kitroom categories/items.
/tkitsadmin reload
Reloads all configuration files.
/tkitsadmin migrate [--all | ]
Migrate kit data between storage backends.
/tkitsadmin kit clear
Forcefully clears a player's specified kit.
/tkitsadmin kit preview
Opens a preview GUI of a player's kit for the admin.
/tkitsadmin give regearitem
Gives the specified player a Regear Box item.
/tkitsadmin cooldown reset
Resets specific or all command cooldowns for a player.

Permissions Overview

  • tkits.use – Use the main /kit command and GUI
  • tkits.load.1 ... tkits.load.7 – Load specific kits
  • tkits.regear – Use the /regear command
  • tkits.arrange – Use the /arrange command
  • tkits.kit.share – Share/import kits
  • tkits.kitroom.use – Use the Kitroom
  • tkits.kitroom.admin – Modify Kitroom layout
  • tkits.admin – All admin commands
  • tkits.cooldown.bypass – Bypass all cooldowns
  • tkits.combattag.bypass – Bypass combat tag restrictions

Command Notes

  • All commands support tab completion and context-aware suggestions.
  • Cooldowns and combat tag restrictions are enforced for all relevant commands.
  • Admin commands require tkits.admin permission.
  • Kitroom and global kit features require appropriate permissions.

Configuration Guide

Every configuration option is documented below, with references to the codebase and real-world effects.

Main Config (config.yml)

  • debug: true|false – Enable verbose logging for debugging.
  • storage.type: YAML|MYSQL – Selects storage backend. MySQL settings are under storage.mysql.
  • kits.max_kits_per_player: int – Maximum kits per player (default: 7).
  • kits.save_on_editor_close: true|false – Auto-save kits when closing the editor.
  • kits.clear_kit_requires_confirmation: true|false – Require confirmation before clearing a kit.
  • kits.clear_inventory_on_load: true|false – Clear inventory before applying a kit.
  • kits.load_requires_empty_inventory: true|false – Prevent loading if inventory isn't empty.
  • kits.prevent_load_in_worlds: [worlds] – List of worlds where kit loading is disabled.
  • kits.prevent_save_items: [materials] – List of forbidden items for kits (e.g., BEDROCK, COMMAND_BLOCK).
  • kits.max_total_items_in_kit: int – Max items per kit (-1 for unlimited).
  • sharing.code_length: int – Length of generated share codes.
  • sharing.code_one_time_use: true|false – Share codes are one-time use.
  • sharing.code_expiration_minutes: int – Minutes before share code expires.
  • sharing.allow_sharing_global_kits: true|false – Allow sharing of global kits.
  • combat_tag.enabled: true|false – Enable combat tag system.
  • combat_tag.duration_seconds: int – Duration of combat tag in seconds.
  • combat_tag.blocked_commands: [commands] – Commands blocked during combat.
  • combat_tag.prevent_enderpearl: true|false – Prevent ender pearl use while tagged.
  • cooldowns.kit_load: int – Cooldown for kit load commands (seconds).
  • cooldowns.regear: int – Cooldown for regear command (seconds).
  • cooldowns.arrange: int – Cooldown for arrange command (seconds).
  • regear.box_material: Material – Material for the regear box (default: SHULKER_BOX).
  • regear.box_name: String – Name for the regear box item.
  • arrange.handle_extra_items: KEEP|DROP|MOVE_EMPTY – How /arrange handles extra items.
  • kitroom.require_per_category_permission: true|false – Require per-category permission for kitroom.
  • kitroom.item_take_cooldown_seconds: int – Cooldown between taking kitroom items.
  • sounds: {key: sound} – Sound effects for various actions (see sounds section).
  • messages: {key: message} – All player/admin messages, fully customizable.

GUI Config (gui.yml)

  • titles: – Titles for all GUIs (main menu, kit editor, kitroom, etc.).
  • items: – Definitions for all GUI items, including material, name, lore, slot, and custom model data.
  • category_overrides: – Per-category button appearance and slot overrides for kitroom.
  • button_area_filler, item_area_filler: – Filler items for GUI backgrounds.
  • Navigation: – Back, next, previous, and info buttons for all GUIs.

Kitroom Config (kitroom.yml)

  • categories: – List of kitroom categories (armor, sword, utilities, potions, arrows, others).
  • Each category contains a mapping of slot numbers (0–44) to serialized ItemStacks.
  • Admins can edit kitroom contents in-game and save changes to this file.

Plugin Descriptor (plugin.yml)

  • name: T-Kits
  • main: com.takeda.tkits.TKits
  • api-version: 1.21
  • author: Takeda_Dev
  • depend: [PlaceholderAPI]
  • commands: – All registered commands, descriptions, usage, and aliases.
  • permissions: – All permission nodes, descriptions, and defaults.

Configuration Notes

  • All config files are hot-reloadable via /tkitsadmin reload.
  • YAML and MySQL storage are fully supported and can be migrated between at any time.
  • All messages and sounds are fully customizable for localization and branding.
  • GUI layouts and items can be themed and rearranged via gui.yml.
  • Kitroom categories and items are managed in-game and saved to kitroom.yml.

Developer Resources

T-Kits exposes a robust API for plugin developers, as well as PlaceholderAPI integration and extensible storage/config/event interfaces.

TKitsAPI (Java API)

  • Optional getPlayerKit(UUID playerUUID, int kitNumber) – Get a player's kit (sync, cached for online).
  • CompletableFuture> getPlayerKitAsync(UUID playerUUID, int kitNumber) – Async kit fetch.
  • Map getAllPlayerKits(UUID playerUUID) – Get all kits for a player.
  • boolean loadKitOntoPlayer(Player player, int kitNumber) – Apply a kit to a player (handles cooldowns, validation).
  • int getLastLoadedKitNumber(Player player) – Get the last kit loaded by a player.
  • List getGlobalKits() – List all global kits.
  • boolean isPlayerCombatTagged(Player player) – Check if a player is combat tagged.
  • long getRemainingCombatTagMillis(Player player) – Get remaining combat tag time.
  • ItemStack getRegearTriggerItem() – Get the special regear box item.
  • boolean performRegear(Player player) – Trigger regear for a player.
  • boolean performArrange(Player player) – Trigger inventory arrange for a player.
Usage Example:
RegisteredServiceProvider provider = Bukkit.getServicesManager().getRegistration(TKitsAPI.class);
if (provider != null) {
  TKitsAPI api = provider.getProvider();
  api.loadKitOntoPlayer(player, 1);
}

PlaceholderAPI Integration

  • %tkits_last_loaded_kit% – Last loaded kit number.
  • %tkits_combat_tagged% – "Yes" if player is combat tagged.
  • %tkits_combat_tag_time% – Remaining combat tag time in seconds.
  • %tkits_kit_X_exists% – "Yes" if kit X exists for the player.
  • %tkits_kit_X_is_global% – "Yes" if kit X is global.

Extensibility & Storage

  • Implement StorageHandler to add new storage backends (YAML, MySQL, or custom).
  • All config files are hot-reloadable and can be extended for new features.
  • Event hooks and listener interfaces are available for custom GUIs, kit logic, and more.

Code Example: Custom Plugin Integration

public class MyPluginListener implements Listener {
  @EventHandler
  public void onPlayerJoin(PlayerJoinEvent event) {
    RegisteredServiceProvider provider = Bukkit.getServicesManager().getRegistration(TKitsAPI.class);
    if (provider != null) {
      TKitsAPI api = provider.getProvider();
      api.performRegear(event.getPlayer());
    }
  }
}

SUPPORT & ANSWERS

FAQ

Find answers to common questions about T-Kits compatibility, features, and support.

Is T-Kits compatible with other plugins?

Yes, T-Kits is designed for broad compatibility. It integrates smoothly with common plugins like Vault, PlaceholderAPI, LuckPerms, WorldGuard, and various economy plugins. We follow standard Bukkit API practices to minimize conflicts.

Does T-Kits support multiple languages?

Absolutely! T-Kits includes a robust localization system. All player-facing messages can be translated. Default language files (like English, Spanish, German) are provided, and you can easily create or customize your own `messages.yml`.

How are updates handled?

Premium users receive update notifications in-game and via Discord/email. Updating is simple: replace the `T-Kits.jar` file and restart your server. Configuration files are generally compatible across minor versions, with migration guides provided for major updates.

What's the difference between Free and Premium?

The Free version offers core kit functionality with limits (e.g., max 5 kits). Premium unlocks unlimited kits, the advanced GUI editor, custom command creation, PlaceholderAPI support, cross-server sync options, priority support, developer API access, and all future advanced features.

How can I get support?

We offer support via our Discord server (community & staff help) and a dedicated ticket system for Premium users (guaranteed response times). Comprehensive documentation and a knowledge base are also available on our website.

Is T-Kits optimized for performance?

Yes, performance is a top priority. We utilize asynchronous operations for database interactions, efficient caching strategies, and optimized code structures to ensure minimal impact on server TPS, even on large networks.
